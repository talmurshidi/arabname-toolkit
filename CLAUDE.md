# CLAUDE.md

Guidance for Claude Code (and other agents) working in this repository.

## Overview

ArabName Toolkit is a free, open-source, fully client-side **React + Vite single-page app** for
converting Brill-transliterated Arabic names to Arabic script (with and without diacritics). It
targets academic researchers in Islamic studies, history, prosopography, and textual scholarship.

The app is deployed via GitHub Pages — there is **no backend, no database, no server**. All logic
runs in the browser. Persistence uses `localStorage` only.

The transliteration engine's rules were originally ported directly from a production Electron app
(TabaqatPerfect), and are **load-bearing scholarly convention** — do not casually edit the rules
in `src/core/transliteration/brill/` without understanding the linguistic rationale (see
`docs/METHODOLOGY.md` and ADR-010). The original file, `legacyEngine.mjs`, is retired (no longer
imported by anything) but kept in the repo as the equivalence oracle for
`tests/core/brillEngineParity.test.ts`.

## Project Structure

For a full, always-current map of every file's exported functions, see `PROJECT_TREE.md` —
generated by `npm run docs:tree` and verified by `npm run verify`.

```
src/
  core/         pure functions — transliteration engine (framework-agnostic, Node-safe)
  shared/       types + pure utilities, no imports from any other src/ module
  services/     browser-API wrappers (localStorage, CSV parsing) — no React, no DOM
  ui/
    i18n/       framework-agnostic locale store + Arabic/English string tables
    hooks/      React bindings (useLocale)
    components/ Header, Footer, VirtualKeyboard, AdvancedOptions
    pages/      one component per view: Converter, Batch, Dictionary, Methodology
  App.tsx       view routing (in-memory state, no router — see ADR-002)
  main.tsx      React entry point
```

Always run `npm run verify` before considering a change complete.

**To save tokens/time on routine changes:** for most changes, running `npm test` and
`npm run typecheck` is enough to confirm correctness — those catch logic errors and type
errors, which is what you're actually changing.

- Don't spend a tool call running `format`/`format:check/run docs:tree/run docs:tree:check` yourself
  unless the developer asks for the full gate. Reserve the complete `npm run verify` (format:check + lint + typecheck + test + docs:tree:check + build) for: finishing a multi-file/multi-task change, before handing
  work back as "done", or when explicitly asked to verify. Don't re-run the full gate after
  every small edit within the same task — accumulate changes and verify once at the end.

## Versioning & Releases

The project follows [Semantic Versioning](https://semver.org/) and
[Keep a Changelog](https://keepachangelog.com/en/1.1.0/), pre-1.0 (see semver §4: anything may
change release to release). `CHANGELOG.md`'s `[Unreleased]` section accumulates entries as
changes are made — this is unchanged (see "Keeping Documentation Current" below: every
user-facing change gets a `### Added`/`### Fixed`/`### Changed` entry there as part of making the
change, same as always).

**What's new: don't hand-edit the version number.** Once a unit of work is complete (its
CHANGELOG entries are written and `npm run verify` passes), run the matching release script to
bump the version and close out the changelog section atomically:

```bash
npm run release:patch   # bug fixes, docs, internal-only changes — no new capability
npm run release:minor   # new features, new dictionary/corrections capability, a rule-version
                         # bump that changes existing output — pre-1.0, this covers breaking
                         # changes too, per semver §4
npm run release:major   # reserved for the deliberate 1.0.0 stability milestone — do not use
                         # casually; ask the user first if a change seems to warrant it
```

Each script (`scripts/release.ts`) reads `CHANGELOG.md`'s `[Unreleased]` section, refuses to run
if it's empty (nothing to release), then runs `npm version <type> --no-git-tag-version` (bumping
`package.json` **and** `package-lock.json` together) and rewrites the changelog so those
`[Unreleased]` entries become a new `## [x.y.z] - <date>` heading, leaving a fresh empty
`[Unreleased]` above it. This keeps the three files — version number, lockfile, changelog — from
ever drifting out of sync, which is what had happened before this was added (package.json had
been bumped to 0.2.0 with no matching changelog section; several rounds of work had piled up under
`[Unreleased]` instead of being cut into their own releases).

Run the release script yourself as part of finishing a task, without waiting to be asked — same
trust level as running `npm test`. It only touches the working tree; it does **not** run `git
commit`, `git tag`, or `git push` — those remain a separate, explicit step under the normal
git-commit workflow (confirm with the user before pushing/tagging, same as any other commit).

## Architecture

See `docs/ARCHITECTURE.md` for the full layer diagram and rationale. Key rules:

- **Layering is one-directional:** `ui/` → `services/` → `core/` → `shared/`. A lower layer never
  imports from a higher one. `core/` and `services/` must stay React-free and DOM-light so they
  remain unit-testable in plain Node (Vitest) and reusable if the UI framework ever changes again
  (see ADR-001 — this is exactly what happened once already, and only `ui/` needed touching).
- **No client-side router.** The app is a single Vite entry point (`index.html`); views are
  switched via in-memory React state in `App.tsx` (see ADR-002). Don't add `react-router` or
  similar without a new ADR justifying it.
- **Styling is Tailwind utility classes only** (see ADR-006). Don't add CSS Modules,
  styled-components, or a second styling system. Shared design tokens (colors, fonts) live in the
  `@theme` block of `src/index.css` — add new tokens there, not as one-off hex values in JSX.

### GitHub Pages deployment

`npm run build` produces `dist/`. The GitHub Actions workflow in `.github/workflows/deploy.yml`
runs `verify` then deploys `dist/` to Pages. Set the `VITE_BASE` Actions variable to
`arabname-toolkit` if deploying to a sub-path; leave it unset for a custom domain.

## Transliteration Engine

The engine is layered (highest priority first) and lives entirely under `src/core/transliteration/`:

0. `din31635.ts` — **only runs when the "DIN 31635" scheme is selected.** Converts DIN 31635's
   seven distinctive letters (ǧ š ġ ḏ ṯ ẖ ẗ) to Brill digraphs before anything else runs
   (`dinToBrill`), and converts Brill output back to DIN spelling for display (`brillToDin`).
   Never modifies the engine in `brill/` — see ADR-007.
   0.5. `orthographyNormalization.ts` — always runs, right after `normalizeTransliteratedName()` and
   before dictionary matching. Two position-aware fixes the dictionary/engine word-level
   matching can't express on its own: `applyIbnAlifRule()` (ابن keeps its alif only as the first
   word of a name; medially it's elided) and `applyElidedArticleRule()` (normalises Brill's
   elided-pronunciation "l-"/"'l-" article spelling to "al-"). Never modifies the engine in
   `brill/` — see ADR-008.
1. `corrections.ts` — exact-match, sourced fixes for specific known-degraded input strings.
   Checked first. Add entries only when verified against a citable source.
2. `dictionary.ts` — the raw curated Latin → Arabic compound/name dictionary
   (`TRANSLITERATION_DICTIONARY`) and its pre-built index (`DICTIONARY_BY_FIRST_WORD`).
3. `dictionaryMatcher.ts` — greedy longest-match scan over that data (`dictionaryTransliterate`,
   `reassembleDictionaryTokens`).
4. `brill/` — the character-level Brill → Arabic engine (`engine.ts`, `consonantTable.ts`,
   `consonantTemplates.ts`, `irregularRules.ts`, `sunLetterAssimilation.ts`,
   `bracketArticleFix.ts`, `helpers.ts`, `nameOrder.ts`). Ported from the original
   `legacyEngine.mjs` (now retired — imported only by its parity test, nowhere in `src/`; see
   ADR-010). **Treat this module's rules as load-bearing scholarly convention, not incidental
   code.** Do not "clean up" its rules without understanding why they are there and without
   updating `tests/core/brillEngineParity.test.ts` (see ADR-003, ADR-010). It only ever sees
   Brill spelling — DIN input is normalised to Brill by layer 0 before it gets here.
5. `bracketSanitizer.ts` — repairs stray/unmatched brackets only; well-formed balanced brackets
   and their contents are preserved verbatim (they are genuine scholarly editorial notations).
6. `arabicHouseStyle.ts` — yā-sukūn post-processing: a medial long-ī (ي preceded by kasra, not
   consonantal) takes a sukūn; a word-final one does not.
7. `index.ts` — the public API (`generateArabic`, `generateArabicHarakat`,
   `normalizeTransliteratedName`, `createNameOrder`, `generateTransliteration`, `generateLatin`,
   `looksFullyDiacritized`, `dinToBrill`, `brillToDin`, `containsDin31635Chars`,
   `applyIbnAlifRule`, `applyElidedArticleRule`, `TRANSLITERATION_RULE_VERSION`). **Import only
   from this file, not from sub-modules.**

The primary direction is **Latin → Arabic**, supporting two input schemes — **Brill** and **DIN
31635** (see `TransliterationScheme` in `src/shared/types.ts`) — selectable in the Converter view.
`TransliterationService.convert()` always populates both `brillLatin` and `dinLatin` in its
result, regardless of which scheme was used to type the input, so the UI can show both Latin
forms side by side. A reverse direction, **fully-diacritised Arabic → Latin**, is also supported
via `arabicToLatin.ts` (`generateLatin`) — a character-level inverse of the productive
(non-idiomatic) rules in the legacy engine, with `brillToDin()` applied afterward to also
populate the DIN form. It is deliberately scoped to fully-voweled Arabic input:
`looksFullyDiacritized()` gates it, since undiacritised Arabic cannot be unambiguously
reverse-mapped without a phonological parser. The Converter view (`src/ui/pages/ConverterPage.tsx`)
exposes both directions via a swap control and a scheme selector; it warns and refuses to convert
rather than guess when the Arabic input isn't fully diacritised. This limitation is documented in
`docs/METHODOLOGY.md` and summarised in the in-app Methodology view.

### DIN 31635 vs Brill

DIN 31635 and Brill share most letters; they differ only in seven (ǧ↔j, š↔sh, ġ↔gh, ḏ↔dh, ṯ↔th,
ẖ↔kh, ẗ↔bare "a" — see `docs/METHODOLOGY.md`). `din31635.ts` handles exactly these seven via
literal/regex substitution — it is a small, closed, deterministic mapping, not a general
transliteration scheme. Do not extend it to handle other romanisation systems (IJMES, ALA-LC);
those remain documentation-only pre-normalisation notes (see `docs/METHODOLOGY.md`), per ADR-007.

The Converter's Brill/DIN 31635 scheme selector works in **both** directions (ADR-009):
Latin → Arabic (which scheme the input is typed in) and Arabic → Latin
(`convertReverse(text, preferredScheme)` — which scheme becomes the result's primary `scheme`
field; both `brillLatin` and `dinLatin` are always populated either way). The Converter also
warns (without blocking) when "Brill" is selected but the input contains a DIN 31635-specific
letter (`containsDin31635Chars()`) — there's no reliable check in the other direction, since
Brill's digraphs are also valid input under the DIN scheme.

### Irregular reverse-direction word mapping

`arabicToLatin.ts`'s `IRREGULAR_REVERSE_WORDS` is a small, explicit lookup for forward-engine
lexical substitutions that can't be inverted character-by-character. Currently one entry: بْن/ابْن
→ `ibn` (both the medial and name-initial Arabic spellings map to the same Brill spelling — see
ADR-009). Extend this list only for other specific, verified irregular words (e.g. a future
`Abū`/`ʿAbdallāh` reversal) — each addition needs a test built from **dynamically-generated**
Arabic input (via `generateArabicHarakat()`), not a hand-typed literal (see the testing rule
below).

### Adding dictionary entries

Add to `TRANSLITERATION_DICTIONARY` in `src/core/transliteration/dictionary.ts`. The
`DICTIONARY_BY_FIRST_WORD` index is built automatically at module load time — no manual update
needed. Compound entries (multi-word) are matched greedily: longer sequences win over shorter
prefixes.

### Adding sourced corrections

Add to `TRANSLITERATION_CORRECTIONS` in `src/core/transliteration/corrections.ts`. Key must be
the exact string as a user would type it (case-sensitive, no normalisation). Value must be the
verified Arabic form. Include a comment citing the source.

### Known-degraded candidates

`docs/reference/degraded-transliteration-candidates.md` is a human worklist. Rows there need
verification against a citable source before being moved to `corrections.ts`. **Do not fill in
values by guessing.**

## UI layer conventions

- **Components:** function components + hooks only. One component per file; co-locate small
  presentational helpers (e.g. `OutputCard` in `ConverterPage.tsx`) in the same file as their
  only consumer rather than creating a new shared file for a single-use component.
- **State:** local `useState` per view; there is no global state manager and none is needed at
  this scale (four views, all reading/writing `localStorage` through their respective service).
  If cross-view shared state grows beyond locale + view (both already in `App.tsx`), reach for
  React context before reaching for a state library.
- **Icons:** `lucide-react` only — don't hand-roll new inline SVG icons or add a second icon set.
- **Design tokens:** colors and fonts are Tailwind theme tokens defined once in `src/index.css`'s
  `@theme` block (see ADR-006). Reference them as utility classes (`bg-scholargreen`,
  `text-ink`); don't introduce raw hex values in component JSX.
- **History:** `historyService.push()` is called for **every** conversion — both directions
  (Latin → Arabic and Arabic → Latin) — from the single `convert()` call site in
  `ConverterPage.tsx` (ADR-009). Don't add a second, direction-specific push path; `ConversionOutput`
  (which `HistoryEntry` extends) already carries `direction` and `scheme` for both cases.

## i18n

- Default locale: Arabic (RTL). English toggle available via the header.
- Locale detection: stored `localStorage` preference → `navigator.language` → Arabic fallback.
- The locale store (`src/ui/i18n/index.ts`) is framework-agnostic and independently unit-testable;
  `src/ui/hooks/useLocale.ts` is a thin `useSyncExternalStore` binding so React components
  re-render on locale change without owning the store's state themselves.
- `useLocale()` returns `{ locale, strings, toggle }`; `strings` is the full `UIStrings` object
  for the current locale — destructure only the section(s) a component needs.
- When adding a new UI string: add it to `UIStrings` in `src/ui/i18n/types.ts` first — TypeScript
  will then fail compilation until both `ar.ts` and `en.ts` are updated.
- Arabic script content (dictionary entries, conversion output, etc.) always renders with the
  `.arabic-text` utility class regardless of the active UI locale — Arabic script should never
  fall back to a Latin serif font just because the UI chrome is in English.

## Testing

- Vitest (`npm test`). Tests live in `tests/` mirroring `src/` structure.
- `tests/core/` tests are pure Node — no DOM, no localStorage. Import directly from `src/core/`.
- `tests/services/` tests stub `localStorage` via `vi.stubGlobal('localStorage', ...)`.
- There is intentionally no component-level UI test suite yet; DOM/React interaction is tested
  manually. If a future regression is UI-specific (not core/services), add a component test using
  Vitest + `@testing-library/react` rather than growing manual-only QA — introduce that
  dependency the first time it's actually needed, not preemptively.
- Port any new regression discovered in production to a named Vitest test before or alongside the
  fix.

### Required test coverage for the transliteration engine

For any new dictionary entry or corrections entry, add at minimum:

1. A test that the exact input produces the exact expected Arabic output.
2. A test that the fallback to the legacy engine still works for something _not_ in the
   dictionary.

For any change to `din31635.ts`'s mapping tables, add at minimum:

1. A test that `dinToBrill()` produces the exact expected Brill spelling.
2. A test that feeding the result through `generateArabicHarakat()` produces Arabic **identical**
   to the equivalent hand-written Brill spelling (compared dynamically — see the note below).
3. A round-trip test: `dinToBrill(brillToDin(x)) === x` for the affected case.

**Do not hand-type expected Arabic-script literals for these tests.** Compute the expected value
dynamically (e.g. `generateArabicHarakat('Jamal')`) and compare against it, rather than pasting an
Arabic string you typed by hand — combining-mark ordering is easy to get subtly wrong by eye, and
a hand-typed literal that merely _looks_ right will pass a visual review but fail (or silently
mismatch) at the codepoint level. This bit `tests/core/din31635.test.ts` during development;
comparing dynamically instead is safer and self-updating.

For bracket sanitizer changes: test the stray-repair case, the well-formed-preserve case, and the
round-trip through `generateArabicHarakat`.

## Conventions

- Formatting: Prettier (`.prettierrc.json`). Linting: oxlint (`oxlint.json`) with
  `--deny-warnings`.
- TypeScript strict mode. Two configs: `tsconfig.json` (browser `src` + `tests`, JSX enabled),
  `tsconfig.node.json` (`scripts`).
- Path aliases: `@core/`, `@services/`, `@ui/`, `@shared/` — defined in both `vite.config.ts` and
  `tsconfig.json`. Use them in `src/`; use relative paths in `tests/` and `scripts/`.
- `legacyEngine.mjs` is excluded from oxlint and TypeScript checking — it is retired (ADR-010),
  kept only as the equivalence oracle imported by `tests/core/brillEngineParity.test.ts`. Don't
  import it from anywhere in `src/`.
- `crypto.randomUUID()` — use `generateId()` from `src/shared/utils.ts` everywhere. Do not call
  `crypto.randomUUID()` directly so that test environments with a Web Crypto polyfill only need
  one stubbing point.

## Never do this

- Don't edit rules in `src/core/transliteration/brill/` without first reading
  `docs/METHODOLOGY.md` and understanding the specific scholarly convention a rule encodes (see
  ADR-003, ADR-010) — and update `tests/core/brillEngineParity.test.ts` deliberately (not
  silently) if the change is an intentional behavior fix, since that test asserts equivalence
  with the retired `legacyEngine.mjs`.
- Don't guess values for `docs/reference/degraded-transliteration-candidates.md` entries — they
  must be verified against a citable source before promotion to `corrections.ts`.
- Don't add a backend, database, or analytics/tracking script — this project's entire value
  proposition is "runs fully offline, in your browser, with nothing sent to a server" (ADR-004).
- Don't add a client-side router, a global state library, or a second styling system without a
  new ADR justifying the change (see ADR-002, ADR-006).
- Don't make the engine in `brill/` "DIN-aware" or add DIN 31635 branches inside it — DIN 31635
  support is a pre/post-processing layer (`din31635.ts`) precisely so the Brill engine never
  needs to change (see ADR-007). Don't extend `din31635.ts` into a general-purpose romanisation
  detector for IJMES/ALA-LC either — those remain documentation-only pre-normalisation notes.

## Keeping Documentation Current

When a change alters something this file or `docs/` describes, update the docs in the same
commit:

- **CLAUDE.md**: update Project Structure, Common Commands, Architecture, Transliteration, UI
  layer conventions, or "Never do this" if a top-level module is added/renamed, a script is
  added, an architectural rule changes, or a new bug class is identified.
- **`docs/ARCHITECTURE.md`** and **`docs/adr/`**: add a new ADR for any decision that changes
  layering, the build target, the UI framework, or a prior ADR's tradeoff.
- **`docs/METHODOLOGY.md`**: update if transliteration rules or their documented limitations
  change; keep the in-app `MethodologyPage.tsx` summary consistent with it.
- **`PROJECT_TREE.md`**: self-enforcing (`npm run verify` fails if stale). Run
  `npm run docs:tree` after adding/removing/renaming any exported symbol.
- **`CHANGELOG.md`**: add an entry under "Unreleased" for any user-facing change.
